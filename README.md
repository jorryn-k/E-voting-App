# E-Voting Console Application (Refactored)

## Overview

This project is a refactored version of an **E-Voting Console Application** originally implemented as a single large Python file. The purpose of the refactoring process was to improve the **structure, readability, and maintainability** of the system while ensuring that the behaviour of the program remains exactly the same.

The refactored application follows **modular programming and object-oriented design principles**. The codebase was divided into logical modules that separate responsibilities such as user interface, data models, business logic, and data persistence.

The system simulates a simple electronic voting platform where administrators manage election data and voters can cast votes through a command-line interface.

## System Components

### 1. Main Application

**main.py** is the entry point of the application.
It initializes the system and controls the flow between authentication, dashboards, and other modules.

---

### 2. Constants

The **constants.py** file stores configuration values and reusable constants used throughout the system.
This helps avoid repetition and makes the code easier to maintain.

---

### 3. Persistence Layer

The **persistence.py** module is responsible for handling data storage.
It manages reading and writing election data to the `evoting_data.json` file.

Responsibilities include:

* loading saved data
* saving updated data
* maintaining data consistency

---

### 4. Models

The **models** package contains classes that represent the core entities of the voting system.

Examples include:

* **Candidate** – represents a candidate participating in an election
* **Voter** – represents a registered voter
* **Admin** – represents a system administrator

These classes encapsulate the data associated with each entity.

---

### 5. Managers

The **managers** package contains the main **business logic** of the system.

Managers control operations such as:

* registering candidates
* managing voters
* handling the voting process
* calculating results

By placing logic in managers, the system separates **application logic from the user interface**.

---

### 6. User Interface

The **ui** package handles all interactions with the user.

It includes:

* **admin dashboards**
* **voter dashboards**
* **authentication screens**

These modules are responsible for displaying menus, receiving user input, and presenting results.

---

### 7. Utilities

The **utils** package contains helper functions used across the system.

Examples include:

* formatting console output
* handling user input safely
* displaying tables or menus

These utilities reduce code duplication and improve readability.

---

## Data Storage

Election data is stored in a JSON file:

```
evoting_data.json
```

This file is automatically generated when the program runs and stores information such as:

* registered voters
* candidates
* votes
* election results

---

## Design Principles Applied

### Modularity

The system was divided into smaller modules that each perform a specific role.

### Object-Oriented Programming

Classes were used to represent entities such as candidates, voters, and administrators.

### Separation of Concerns

Different parts of the system were separated into layers:

* **UI layer** – handles user interaction
* **Managers layer** – contains business logic
* **Models layer** – represents system data
* **Persistence layer** – handles data storage

### Maintainability

The modular design makes the code easier to understand, modify, and extend.

---

## How to Run the Application

1. Ensure **Python 3** is installed.
2. Navigate to the project folder.
3. Run the program using:

```
python evoting/main.py
```

The console interface will display login options for administrators and voters.

---

## Conclusion

The refactoring process transformed a large monolithic script into a **structured, modular, and maintainable system**. By applying object-oriented principles and separating responsibilities into different modules, the project now follows good **software engineering practices** and is easier to extend and maintain.
